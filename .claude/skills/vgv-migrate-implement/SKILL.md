# Implement Flutter Feature (FFCA)

This skill implements one feature triple — `{feature}_domain`, `{feature}_data`, `{feature}_presentation` — following the project's Feature-First Clean Architecture, VGV coding standards, and the specification in `docs/migration-manifest.yaml`.

## Usage

```
/vgv-migrate-implement <feature-id> [--manifest=path] [--dry-run] [--serial] [--figma=url]
```

| Argument | Description |
|---|---|
| `feature-id` | Feature ID as it appears in `migration-manifest.yaml` (required) |
| `--manifest` | Path to manifest (default: `docs/migration-manifest.yaml`) |
| `--dry-run` | Run pre-flight checks only, print plan, do not write code |
| `--serial` | Skip sub-agent parallelism, implement all layers sequentially |
| `--skip-review` | Skip Phase 9 post-implementation review entirely |
| `--figma` | One or more Figma URLs (repeatable). Each URL points to a screen frame. The skill extracts `fileKey` and `nodeId` and fetches design context via Figma MCP. |

### Examples

```bash
/vgv-migrate-implement cart
/vgv-migrate-implement auth --dry-run
/vgv-migrate-implement loyalty --serial
/vgv-migrate-implement product --manifest=docs/migration-manifest.yaml
/vgv-migrate-implement settings --figma=https://www.figma.com/design/03eW3Tr0cqi5XcL5coZk7J/-WIP--SC-Flutter-App-Experience?node-id=12454-16735
/vgv-migrate-implement settings \
  --figma=https://...?node-id=12454-16735 \
  --figma=https://...?node-id=12454-16800
```

## Workflow

### Phase 0: Read context

Before any other action, read:
1. `CLAUDE.md` — project standards, coverage requirements, architecture rules, package naming conventions
2. `docs/migration-manifest.yaml` (or `--manifest` path) — locate the feature by ID, extract its full slice
3. `references/FFCA_PATTERNS.md` — code templates and conventions to follow during generation
4. **Native analysis document** — read `docs/features/{feature}/ANALYSIS.md` (generated by the `vgv-migrate-analyze` skill) to extract business logic, UI structure, validation rules, state transitions, and data flow patterns that the manifest alone cannot capture
5. **Figma design references** — if `--figma` URLs were provided, parse each URL to extract `fileKey` and `nodeId` (pattern: `figma.com/design/:fileKey/...?node-id=:nodeId`, convert `-` to `:` in nodeId). Call `get_design_context` and `get_screenshot` for each. Store results as `figma_context_map` keyed by nodeId. Match each Figma node to a screen name using the node's name from the response. If Figma MCP is unavailable or a call fails, emit a soft warning per URL and proceed without that screen's design data

If `CLAUDE.md` or the manifest is missing, **halt immediately** with a clear error.

#### Native parity analysis (Phase 0, step 4)

The manifest captures screen names, API endpoints, and model dependencies, but does not encode business logic, validation rules, state machine behavior, or UI interaction patterns. The per-feature native analysis document fills these gaps.

**Primary input: `docs/features/{feature}/ANALYSIS.md`**

This file is generated by the `vgv-migrate-analyze` skill and contains a comprehensive analysis of the native implementations for this feature, including:
- Screen inventory with ViewModel/View file paths and responsibilities
- Business logic rules extracted from native ViewModels/UseCases
- State transitions and their triggers
- Validation rules (field constraints, format checks)
- Data transformation logic (computed properties, formatting)
- UI component hierarchy and interaction patterns
- Data flow details (caching, pagination, error handling)
- Platform differences between iOS and Android

**Procedure:**

1. **Read `docs/features/{feature}/ANALYSIS.md`** — this is the primary source of native parity data. Parse the structured sections to extract business logic, state transitions, validation rules, UI patterns, and data flow details.

2. **Fallback: read native source files directly** — only if `ANALYSIS.md` does not exist for this feature. Use the feature's `ios_screens` and `android_screens` arrays from the manifest to locate source files in the native codebases (symlinked at `native/{project}-ios/` and `native/{project}-android/`, where `{project}` is defined in the manifest's `project_name` field). See `references/REFERENCE.md` for the full native file discovery procedure.

3. **Reconcile manifest vs analysis** — compare the manifest's `api_dependencies`, `data_model_dependencies`, and screen lists against what the ANALYSIS.md (or live native reading) reveals. Surface any gaps:
    - API endpoints used in native but missing from the manifest
    - Models with fields present in native but absent from the manifest
    - Screens or flows present in native but not listed in the manifest
    - Business logic (validation, formatting, computed values) with no manifest representation

**Soft failures (do not halt):**
- `docs/features/{feature}/ANALYSIS.md` not found AND native symlinks not present → skip native analysis, add ⚠ to pre-flight: "Native analysis not available — generating from manifest only, manual parity review recommended"
- A screen listed in the manifest cannot be found in the analysis or native source → note in summary, proceed with manifest-only data for that screen
- Native code uses patterns not applicable to Flutter (e.g., iOS Combine publishers) → translate the intent, not the mechanism

### Phase 1: Generate NATIVE_PARITY.md (main agent, serial)

After completing native parity analysis in Phase 0, write a persistent checklist to `docs/features/{feature}/NATIVE_PARITY.md`. This file is committed to the branch and read by all sub-agents as hard requirements.

**Purpose:** The in-memory parity summary from Phase 0 is ephemeral — sub-agents need a durable, structured artifact they can read from disk. NATIVE_PARITY.md converts the analysis into a machine-checkable format.

**Format:**

```markdown
# {Feature} — Native Parity Checklist

Source: docs/features/{feature}/ANALYSIS.md | live native file reading
Generated by vgv-migrate-implement Phase 1

## Business Logic Rules

- [ ] {BL-01}: {description} (source: {iOS ViewModel / Android ViewModel / both})
- [ ] {BL-02}: {description}
...

## State Transitions

### {ScreenName}
- [ ] {ST-01}: {State A} → {State B} on {trigger} (source: {file})
- [ ] {ST-02}: {State A} → {State C} on {trigger}
...

## Validation Rules

- [ ] {VR-01}: {field}: {constraint} (source: {file})
- [ ] {VR-02}: {field}: {constraint}
...

## UI Patterns

### {ScreenName}
- [ ] {UI-01}: {component/interaction pattern} (source: {file})
- [ ] {UI-02}: {conditional rendering rule}
...

## Data Flow

- [ ] {DF-01}: {endpoint}: {caching/pagination/error handling detail}
- [ ] {DF-02}: {endpoint}: {detail}
...

## Navigation

- [ ] {NV-01}: {ScreenA} → {ScreenB} with params: {params} (source: {file})
- [ ] {NV-02}: {return behavior description}
...

## Figma Design References

### {ScreenName}
- [ ] {FD-01}: {layout/spacing/color/typography observation} (figma: {file_key}/{node_id})
- [ ] {FD-02}: {observation}
...

(Section omitted if no --figma URLs were provided)

FD-xx items are derived from Figma `design_context`. They complement UI-xx items: FD-xx = what it SHOULD look like (design), UI-xx = what it DOES look like (native). When they conflict on visual appearance, FD-xx takes precedence.

## Platform Differences

- [ ] {PD-01}: {difference and chosen Flutter approach}
...

## Manifest Gaps

Items found in native analysis but missing from migration-manifest.yaml:
- [ ] {MG-01}: {description of gap}
...
```

Each item has a unique ID (e.g., BL-01, ST-01) so sub-agents and reviewers can reference specific requirements. Items start unchecked — Phase 9 Reviewer D checks them against the generated code.

If native analysis was skipped (no ANALYSIS.md and no native symlinks), write a minimal NATIVE_PARITY.md noting that native parity data is unavailable and all items are manifest-derived.

**Commit NATIVE_PARITY.md to the branch before proceeding to Phase 2.**

### Phase 2: Pre-flight checks

Run all checks before writing any code. Categorize results as hard failures (halt) or soft warnings (proceed with fallback).

#### Hard failures (halt)
- Manifest does not contain the requested `feature-id`
- `CLAUDE.md` not found
- Target package paths already exist (e.g., `features/cart/cart_domain`) — prevent overwrite
- Workspace `pubspec.yaml` not found or not writable

#### Soft warnings (proceed with fallback)
- A cross-feature `{dep}_domain` package is referenced in the manifest but not present in the workspace → generate interface stub + TODO
- A shared package barrel file (`shared/{package}/lib/{package}.dart`) cannot be read → flag in summary, generate stubs for all references to that package

#### Pre-flight output

Print a structured plan to the developer and **wait for explicit confirmation** before proceeding:

```
Feature: {feature-id} (effort: {size}, complexity: {score}/5)

Packages to create:
  features/{feature}/{feature}_domain        (Dart)
  features/{feature}/{feature}_data          (Dart)
  features/{feature}/{feature}_presentation  (Flutter)

Cross-feature dependencies:
  {✓|✗|⚠} {dep}_domain — {found at path | STUB: not found}

Shared packages:
  {✓|⚠} {package} — {barrel found, {n} exported symbols | barrel NOT FOUND}

Native parity analysis:
  {✓|⚠} Source: docs/features/{feature}/ANALYSIS.md {found | NOT FOUND — using live native reading | NOT AVAILABLE}
  NATIVE_PARITY.md: {n} checklist items ({n} business logic, {n} state transitions, {n} validation, {n} UI, {n} data flow)
  Manifest gaps: {n} items found in native but missing from manifest
  Platform differences noted: {n}

Figma design references:
  {✓|⚠} {node_name} ({node_id}) — {design_context fetched | MCP failed: {error}}
  ...
  (omitted if no --figma URLs provided)

Complexity flags: (from manifest + native analysis)

{list of TODOs that will be generated due to missing deps/symbols}

Proceed? [y/n]
```

If `--dry-run` is set, print this plan and exit. Do not proceed even if the developer replies "y".

### Phase 3: Domain scaffold (main agent, serial)

Create the `{feature}_domain` package. This phase is always serial — Data and Presentation sub-agents depend on a stable domain.

**Package structure to create:**

```
features/{feature}/{feature}_domain/
├── pubspec.yaml
├── lib/
│   ├── {feature}_domain.dart          # barrel file
│   ├── models/
│   │   └── {model}.dart               # one file per domain model
│   ├── repositories/
│   │   └── i_{feature}s_repository.dart
│   └── use_cases/                     # only if needed (see rules below)
│       ├── {verb}_{noun}_command.dart
│       └── {verb}_{noun}_query.dart
└── test/
    └── (empty — domain tests written by sub-agent C)
```

**Domain generation rules:**

- Models: pure Dart, no `Model` or `Entity` suffix, `const` constructors, immutable fields. Read `docs/features/{feature}/NATIVE_PARITY.md` to ensure all fields present in native models are accounted for — if a native model has fields not listed in the manifest (flagged as MG-xx items), add them. If a field's purpose is unclear, add it with a `// TODO` comment noting the native source
- Repository interface: `abstract interface class I{Feature}sRepository`, one method per CRUD/watch operation defined in the manifest. Cross-reference with NATIVE_PARITY.md DF-xx items to catch operations the manifest may have missed (e.g., a `deleteX` or `watchX` present in native but absent from the manifest's `api_dependencies`)
- Use cases: only generate if the manifest lists multiple repository dependencies for this feature, or if NATIVE_PARITY.md BL-xx items show cross-repo logic in ViewModels/UseCases; otherwise skip and add a TODO
- Commands: named `{Verb}{Noun}Command`, method named `execute`
- Queries returning Future: named `{Verb}{Noun}Query`, method named `get`
- Queries returning Stream: named `{Verb}{Noun}Query`, method named `watch`
- Never use callable classes (`call` method)
- Cross-feature deps: if `{dep}_domain` exists in workspace, add it to `pubspec.yaml`; if not, generate a minimal stub interface under `lib/stubs/` with a TODO comment

**Commit domain scaffold to branch `feature/{feature}-domain` before proceeding.**

### Phase 4: Register packages in workspace (main agent, serial)

Before launching sub-agents:

1. Add all three packages to the workspace `pubspec.yaml` (even though data and presentation directories don't exist yet — sub-agents will create them)
2. Create empty stub `pubspec.yaml` files at the data and presentation paths so the workspace resolves cleanly

### Phase 5: Launch sub-agents in parallel

Launch three sub-agents concurrently. Each sub-agent operates in an isolated git worktree (using the Agent tool's `isolation: "worktree"` parameter), which gives it a full copy of the repo without branch management overhead. Sub-agents read the domain scaffold from the committed code and never modify the domain package.

Pass each sub-agent:
- The full feature manifest slice
- The committed domain package path
- Relevant shared package barrel file exports and source file paths
- The FFCA_PATTERNS.md reference
- The relevant sections of CLAUDE.md (coverage requirements, naming rules, test conventions)
- **The `docs/features/{feature}/NATIVE_PARITY.md` file path** — sub-agents must read this file and treat each checklist item as a hard requirement. This is critical for ensuring the generated code matches the native app's behavior, not just its API surface
- **Figma design context** (if `--figma` was provided and data was fetched) — per-screen `design_context` JSON, `screenshot` reference, and `file_key`/`node_id` for additional MCP calls. Passed only to Sub-agent B

See `references/REFERENCE.md` for detailed per-sub-agent instructions.

#### Sub-agent A: Data layer (worktree)

**Before generating anything**, check whether a pre-generated API client package already covers this feature's endpoints. The project uses a Swagger-to-Dart generator that produces full client packages (e.g. `shared/{project}_api_v2`, `shared/{project}_api_v3`). These packages already contain DTOs and remote method implementations — do not regenerate them.

See `references/REFERENCE.md` Sub-agent A section for the full discovery and resolution logic.

Creates `{feature}_data` package. The exact structure depends on what the pre-generated client provides:

**When a pre-generated client covers this feature's endpoints (common case):**

```
{feature}_data/
├── pubspec.yaml
├── lib/
│   ├── {feature}_data.dart
│   ├── mappers/
│   │   └── {model}_mapper.dart        # DTO (from generated client) → domain model
│   └── repositories/
│       └── {feature}s_repository.dart # implements I{Feature}sRepository
└── test/
    ├── mappers/
    │   └── {model}_mapper_test.dart
    └── repositories/
        └── {feature}s_repository_test.dart
```

**When no generated client covers the endpoints (fallback — generate manually):**

```
{feature}_data/
├── pubspec.yaml
├── lib/
│   ├── {feature}_data.dart
│   ├── data_sources/
│   │   ├── {feature}_remote_data_source/
│   │   │   ├── {feature}_remote_data_source.dart
│   │   │   └── dtos/
│   │   │       └── {dto}.dart              # json_serializable annotated
│   │   └── {feature}_local_data_source/    # only if manifest has local storage
│   │       └── {feature}_local_data_source.dart
│   ├── mappers/
│   │   └── {model}_mapper.dart
│   └── repositories/
│       └── {feature}s_repository.dart
└── test/
    ├── data_sources/
    │   └── {feature}_remote_data_source_test.dart
    ├── mappers/
    │   └── {model}_mapper_test.dart
    └── repositories/
        └── {feature}s_repository_test.dart
```

Coverage target: 100%+. Use `mocktail` for mocking. Follow CLAUDE.md test conventions.

#### Sub-agent B: Presentation layer (worktree)

Creates `{feature}_presentation` package:

```
{feature}_presentation/
├── pubspec.yaml
├── lib/
│   ├── {feature}_presentation.dart
│   └── {screen_name}/
│       ├── bloc/
│       │   ├── {screen}_cubit.dart        # or _bloc.dart if event-driven
│       │   └── {screen}_state.dart
│       ├── views/
│       │   └── {screen}_screen.dart
│       ├── widgets/                       # screen-local widgets only
│       └── {screen}_module.dart           # exports all public symbols
└── test/
    ├── {screen_name}/
    │   ├── bloc/
    │   │   └── {screen}_cubit_test.dart
    │   └── views/
    │       └── {screen}_screen_test.dart
    └── helpers/
        └── pump_app.dart
```

Coverage target: 100%+. Use `bloc_test` for cubit/bloc tests, `mocktail` for mocking, `flutter_test` for widget tests. Read ui_kit source files for component constructor signatures before generating widget code. If a required ui_kit component is not found in the barrel file exports, use a placeholder `Container` or `Text` widget with a structured TODO.

**Native parity requirements for presentation:**

Sub-agent B must read `docs/features/{feature}/NATIVE_PARITY.md` and address every relevant checklist item:
- **ST-xx (State Transitions):** Ensure the cubit/bloc handles all states present in the native ViewModel (e.g., if native has a `Saving` state, the cubit must emit a corresponding state)
- **UI-xx (UI Patterns):** Replicate the component hierarchy — match the native layout structure with Flutter equivalents (e.g., native `LazyColumn` → Flutter `ListView`, native `NavigationLink` → Flutter `Navigator.push`)
- **VR-xx (Validation Rules):** Implement client-side validation matching native behavior
- **BL-xx (Business Logic):** Implement any computed properties or derived display values that exist in native ViewModels
- **NV-xx (Navigation):** Match navigation patterns, parameter passing, and return-from-screen behavior

For each NATIVE_PARITY.md item addressed, add a code comment referencing the item ID: `// Ref: {ID} — {brief description}`

**Figma design requirements for presentation:**

When Figma data is available for a screen (via `--figma`), Sub-agent B must use it to guide visual layout decisions. Priority: FD-xx (design truth for visuals) > UI-xx (behavioral truth) > FFCA_PATTERNS.md (code structure).

- Map Figma auto-layout to Column/Row/Stack
- Map Figma spacing to AppSpacing tokens (or exact values with `// Figma: {n}px` comment)
- Map Figma colors to theme tokens
- Map Figma typography to TextTheme
- Add `// Ref: FD-xx` comments for each design item addressed

If no `--figma` URLs were provided, fall back to UI-xx items only (current behavior).

Writes tests for the domain scaffold:

```
{feature}_domain/test/

│   └── i_{feature}s_repository_test.dart  # contract tests via mock impl
└── use_cases/
```

Coverage target: 100%+.

### Phase 6: Merge sub-agent worktrees (main agent)

After all three sub-agents complete, each returns a worktree path and branch with its changes:

1. Merge the data layer worktree branch → current branch
2. Merge the presentation layer worktree branch → current branch

### Phase 9: Post-implementation review (main agent or agent team)

Phase 9 runs after Phase 8 unless `--skip-review` is passed. It never blocks the skill — all findings are advisory. The output is `docs/features/{feature}/REVIEW.md`.

#### Detecting agent team availability

Check for the environment variable before deciding which mode to use:

```bash
# Agent teams available if this is set to "1"
echo $CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS
```

If the variable is set to `"1"`, use **agent team mode**. Otherwise, use **sequential fallback mode**. Print which mode is active at the start of Phase 9.

#### Agent team mode

Spawn a review team with four reviewers. Each reviewer operates independently, reading only the committed `feature/{feature}` branch. Pass all four the same base context: the three package paths, the manifest feature slice, CLAUDE.md, and `docs/features/{feature}/NATIVE_PARITY.md`.

The key advantage over sub-agents here is peer communication: each reviewer can message the others when a finding in their domain has implications for another reviewer's criteria. For example, if Reviewer A finds that `cart_data` imports directly from `product_data` (a boundary violation), it should message Reviewer B, since any tests mocking that import are also suspect.

**Reviewer A: FFCA boundary compliance**

Checks that the dependency graph is correct:
- Presentation imports only from `{feature}_domain` (never from `{feature}_data`)
- Data imports only from `{feature}_domain` (never from `{feature}_presentation`)
- No cross-feature `_data` package imports
- No imports from `apps/`
- Stub files are in the correct location (`lib/stubs/`, not mixed into production code)
- Barrel files export only the intended public surface (no accidental leaking of internal classes)

If a boundary violation is found that affects test imports, **message Reviewer B** with the specific file and import path.

**Reviewer B: Test coverage and quality**

Checks test files across all three packages:
- Every public class has at least one test file
- Mapper tests cover both the happy path and null/optional field cases
- Repository tests cover both success and exception propagation paths
- BLoC/Cubit tests cover all state transitions defined in the state class
- No test file imports from `{feature}_data` in the domain or presentation test suites (domain tests must be pure)
- No tests that only assert `isA<SomeState>()` without checking specific props (shallow assertions)

If a test file's mocks reference an import that Reviewer A has flagged as a boundary violation, **message Reviewer A** to correlate the finding.

**Reviewer C: Stub inventory and actionability**

Scans all `// TODO(vgv-migrate-implement):` comments across the three packages:
- Verifies each stub has all four required fields: package, reason, action, tracking reference
- Identifies stubs that could be resolved immediately with existing workspace packages (i.e., the dep exists but the stub was generated due to a timing issue)
- Flags any stubs where the action is vague ("implement when ready") vs. specific ("add X method to Y shared package")
- Counts stubs by category: missing shared API method, missing ui_kit component, missing cross-feature dep, unclear business logic

If a stub is resolvable immediately (the package is now present), **message Reviewer A** to check whether the stub creates a boundary issue in its current form.

**Reviewer D: Native parity compliance**

Reads `docs/features/{feature}/NATIVE_PARITY.md` and checks each checklist item against the generated code:
- For each **BL-xx** item: verify the business logic rule is implemented in the relevant cubit/repository
- For each **ST-xx** item: verify the state transition exists in the cubit's state class and is triggered correctly
- For each **VR-xx** item: verify the validation rule is implemented and produces appropriate error states
- For each **UI-xx** item: verify the UI pattern is present in the screen widget (component hierarchy, conditional rendering, interactions)
- For each **NV-xx** item: verify the navigation pattern matches (push/replace, parameters, return behavior)
- For each **DF-xx** item: verify the data flow detail is reflected in the repository or cubit (pagination, caching, error mapping)
- For each **MG-xx** (manifest gap) item: verify it was addressed or has a TODO
- For each **FD-xx** item: verify the Figma layout structure, spacing, and token usage are reflected in the screen widget. Blocking if widget tree structure contradicts Figma layout (e.g., vertical vs horizontal). Action required if spacing/color differs significantly from Figma design. Informational for minor token rounding differences

Classification:
- **Blocking**: A native behavior with user-facing impact (validation rule, state transition, navigation flow) that is completely missing from the generated code
- **Action required**: A native behavior that is partially implemented or has a TODO but could be completed now
- **Informational**: Platform-specific behaviors that were intentionally simplified, or minor computed properties that can be deferred

If Reviewer D finds a missing state transition that should be tested, **message Reviewer B** to check whether the corresponding test gap exists.

After all four reviewers complete their individual passes, the team lead synthesizes findings into `REVIEW.md`.

#### Sequential fallback mode

The main agent performs the same four reviews in sequence (boundary → coverage → stubs → native parity), applying the identical criteria. Peer communication is replaced by a single accumulated findings list. Print `[Review mode: sequential fallback — CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS not set]` before starting.

```markdown
# {Feature} — Implementation Review

Generated by vgv-migrate-implement Phase 9.
Review mode: agent team | sequential fallback
Date: {ISO date}

## Summary

|---|---|---|---|

| Test coverage | {n} | {n} | {n} |
| Stub inventory | {n} | {n} | {n} |
```

```dart
// TODO(vgv-migrate-implement): [cart_data] TalonOneDataSource.applyOffer
```

## Boundaries the skill never crosses

| Boundary | Reason |
|---|---|
| Never modifies `apps/` | Routing changes break the app shell; wiring is a developer step |
| Never modifies `shared/` | Shared package changes need their own PR |
| Never modifies other features' packages | Bounded context — cross-feature stubs are the correct pattern |
| Never auto-resolves merge conflicts | Conflicts indicate a Phase 4 failure; developer must inspect |
| Never skips the pre-flight gate | Even in `--serial` mode, confirmation is always required |
| Never makes Phase 9 findings blocking by default | Review findings are advisory; `REVIEW.md` is the output, not a gate |
