# Implementation Reference Guide

Detailed instructions for each phase and sub-agent. Read this alongside SKILL.md.

---

## Phase 0: Reading context

### Reading CLAUDE.md

Extract and retain:
- Package naming conventions (e.g., prefix, casing)
- Minimum code coverage threshold (typically 100%)
- Required linting rules or `analysis_options.yaml` conventions
- Any project-specific patterns that override FFCA defaults (e.g., a custom base BLoC class)
- `build_runner` invocation command (may differ per project)
- Git branch naming convention

### Reading the manifest

Locate the feature by `id` in the `features` array. Extract:

| Field | Used for |
|---|---|
| `name` | Human-readable label in summaries |
| `effort_size` | Surface in pre-flight; L/XL triggers extra TODO detail |
| `complexity` | Surface in pre-flight; 4-5 triggers confirmation prompts |
| `api_dependencies` | Enumerate required API endpoint IDs ŌåÆ look up in `api.endpoints` |
| `data_model_dependencies` | Enumerate required model IDs ŌåÆ look up in `data_models` |
| `platform_api_dependencies` | Enumerate platform API IDs ŌåÆ look up in `platform_apis`; flag if `flutter_solution.type == platform_channel` |
| `feature_dependencies` | Cross-feature domain deps ŌåÆ check workspace for presence |
| `migration_notes` | Surface verbatim in pre-flight and WIRING.md |
| `parity_notes` | Surface in WIRING.md; flag behavioral differences between native platforms |

### Reading native analysis

After extracting the manifest slice, read the native analysis for this feature. The goal is to capture business logic, validation, state management, and UI patterns that the manifest's declarative format cannot express.

#### Step 1: Read `docs/features/{feature}/ANALYSIS.md` (primary source)

This file is generated by the `vgv-migrate-analyze` skill and contains a structured analysis of the native implementations. It typically includes:

- **Screen inventory**: Each screen with its iOS/Android file paths, ViewModel responsibilities, and state management patterns
- **Business logic**: Validation rules, computed properties, formatting logic, side effects
- **Data flow**: API calls, caching strategies, pagination patterns, error handling
- **UI patterns**: Component hierarchy, conditional rendering, user interactions
- **Platform differences**: Behavioral divergences between iOS and Android

Parse this document to extract structured data for NATIVE_PARITY.md generation (Phase 1).

#### Step 2: Fallback ŌĆö read native source files directly

Only if `docs/features/{feature}/ANALYSIS.md` does not exist. Use the feature's `ios_screens` and `android_screens` arrays from the manifest to locate source files in the native codebases (symlinked at `native/{project}-ios/` and `native/{project}-android/`, where `{project}` is the manifest's `project_name` field).

Search strategy ŌĆö for each screen name in the manifest, search using Glob/Grep:

**iOS** (search within `ios.project_structure.source_directories`):
```
native/{project}-ios/**/Settings*.swift
native/{project}-ios/**/SettingsView*.swift
native/{project}-ios/**/SettingsViewModel*.swift
native/{project}-ios/**/{Feature}Service.swift
native/{project}-ios/**/{Feature}Repository.swift
```

**Android** (search within `android.project_structure.source_directories`):
```
native/{project}-android/**/Settings*.kt
native/{project}-android/**/SettingsViewModel*.kt
native/{project}-android/**/{Feature}Repository*.kt
native/{project}-android/**/{Feature}UseCase*.kt
```

If native symlinks are broken or the directory doesn't exist, emit a soft warning and continue with manifest-only data.

When reading native files directly, extract in priority order:

1. **ViewModels / UseCases** (highest priority) ŌĆö business logic:
    - Public methods and parameters ŌåÆ map to cubit methods
    - State properties and observable values ŌåÆ map to state class fields
    - Validation logic ŌåÆ replicate in cubit or form validation
    - Error handling patterns, side effects, computed properties

2. **Views / Screens / Composables** ŌĆö UI structure:
    - Component hierarchy, conditional rendering, user interactions
    - Navigation flows with parameters

3. **Services / Repositories** ŌĆö data layer:
    - Method signatures, request construction, caching, offline behavior

4. **Models / DTOs** ŌĆö supplementary fields, computed properties, encoding logic

#### Step 3: Reconcile manifest vs native analysis

Compare manifest data with ANALYSIS.md (or live native reading) findings:

- API endpoints used in native but missing from manifest ŌåÆ flag as MG-xx items
- Models with fields present in native but absent from manifest ŌåÆ flag as MG-xx items
- Screens/flows present in native but not listed in manifest ŌåÆ flag as MG-xx items

#### Step 4: Build NATIVE_PARITY.md

Use the analysis data to generate the `docs/features/{feature}/NATIVE_PARITY.md` checklist as described in SKILL.md Phase 1. Each item must be specific and actionable:

Good: `"BL-01: Profile save validates: firstName non-empty, lastName non-empty, email matches regex, phone optional but must be 10+ digits if provided (source: ProfileViewModel.swift)"`

Bad: `"BL-01: Profile has validation"` (too vague to implement from)

Good: `"DF-01: TransactionHistory uses offset pagination: pageNumber (1-based), pageSize (default 20), response includes totalPages (source: OrderHistoryViewModel.kt)"`

Bad: `"DF-01: Transactions are paginated"` (doesn't tell the sub-agent how)

### Reading shared package APIs

For each shared package referenced by the feature (via `api_dependencies` or ui_kit usage):

1. Read the barrel file at `shared/{package}/lib/{package}.dart`
2. Follow the barrel's exports to identify the re-exported files (e.g. `export 'src/export.dart'`, `export 'src/widgets/widgets.dart'`)
3. Read those intermediate barrel/export files to build the full list of exported symbols (class names, client names, model names)
4. Build a **capability map** of available symbols: `{ "ClassName": "src/path/to/file.dart" }`

When constructor signatures or method signatures are needed (e.g. to generate mapper code or widget usage):
1. Use the capability map to locate the source `.dart` file for the specific class
2. Read that source file to extract the exact constructor parameters, method signatures, and types
3. Only read source files that are actually needed ŌĆö do not read the entire package

If the barrel file cannot be read (package missing or empty):
1. Flag as a soft warning in the pre-flight output
2. Generate stubs with TODOs for all references to that package

## Phase 2: Pre-flight detail

### Cross-feature dependency resolution

For each `feature_dependency` in the manifest:

```
1. Check workspace/pubspec.yaml for the dep package name
2. If found: resolve path, confirm lib/{dep}_domain.dart barrel exists
3. If not found:
   a. Add "ŌÜĀ {dep}_domain ŌĆö NOT FOUND in workspace" to pre-flight
   b. Note: will generate a minimal stub interface at
      features/{feature}/{feature}_domain/lib/stubs/{dep}_domain_stub.dart
   c. Add TODO entry to the list
```

Stub interface format (generated in Phase 3):

```dart
// TODO(vgv-migrate-implement): [{feature}_domain] {DepDomain} stub
// Reason: {dep}_domain not found in workspace at pre-flight time.
// Action: replace this stub with the real package once {dep} is implemented.
// Add to pubspec.yaml: {dep}_domain: {path: ../../{dep}/{dep}_domain}

// ignore: avoid_classes_with_only_static_members
abstract interface class I{Dep}Repository {
  // Stub: add methods as {dep}_domain is defined
}
```

### Platform API flags

For each `platform_api_dependency`:
- If `flutter_solution.type == "platform_channel"` or `"not_supported"`: add a prominent ŌÜĀ flag to pre-flight, add a TODO
- If `flutter_solution.recommended_package` is set: add it to the relevant layer's `pubspec.yaml` dependencies automatically

### Effort/complexity prompts

If `effort_size` is `L` or `XL`, or `complexity` is `4` or `5`, append to the pre-flight plan:

```
ŌÜĀ This feature is rated {effort_size} effort / complexity {n}/5.
  The following aspects may require manual review:
  - {migration_notes from manifest}
  - {parity_notes from manifest}
  Stubs and TODOs will be generated for ambiguous areas.
```

## Phase 3: Domain scaffold detail

### pubspec.yaml template

```yaml
name: {feature}_domain
description: Domain layer for the {feature} feature.
version: 0.1.0+1
publish_to: none

environment:
  sdk: ">=3.0.0 <4.0.0"

dependencies:
  # Cross-feature domain deps (if any)
  # {dep}_domain:
  #   path: ../../{dep}/{dep}_domain

dev_dependencies:
  very_good_analysis: ^{version}  # read from other packages in workspace
  test: ^1.25.0
```

### Domain model rules

- One file per model, named `{model_name_snake_case}.dart`
- `const` constructors
- Named required parameters only (no positional)
- Immutable: all fields are `final`
- No `copyWith`, `toJson`, `fromJson` ŌĆö those belong in data layer DTOs
- If the manifest `data_models` entry has `parity_issues`, add a comment per issue
- **Native parity**: read `docs/features/{feature}/NATIVE_PARITY.md` and cross-reference MG-xx items for model fields present in native but missing from the manifest ŌĆö add them. If NATIVE_PARITY.md lists computed properties (BL-xx items) that derive from stored fields, document them as a comment (they may become cubit-level logic)

```dart
/// {description from manifest or inferred from native analysis}
class {ModelName} {
  const {ModelName}({
    required this.{field},
    // ...
  });

  final {Type} {field};
  // ...

  @override
  bool operator ==({Object} other) =>
      identical(this, other) ||
      other is {ModelName} && {fields comparison};

  @override
  int get hashCode => Object.hash({fields});

  @override
  String toString() => '{ModelName}({fields})';
}
```

### Repository interface rules

- Named `I{Feature}sRepository` (plural, prefixed with `I`)
- `abstract interface class` ŌĆö not `abstract class`
- Methods derived from manifest `api_dependencies`: map HTTP verbs to CRUD semantics

| HTTP | Repository method pattern |
|---|---|
| GET (single) | `Future<{Model}> get{Model}ById(String id)` |
| GET (list) | `Future<List<{Model}>> get{Model}s()` |
| GET (stream) | `Stream<{Model}> watch{Model}ById(String id)` |
| POST | `Future<void> create{Model}({Model} model)` |
| PUT/PATCH | `Future<void> update{Model}({Model} model)` |
| DELETE | `Future<void> delete{Model}(String id)` |

**Native parity for repository interfaces:** After generating the interface from the manifest's `api_dependencies`, cross-reference with `docs/features/{feature}/NATIVE_PARITY.md` DF-xx and MG-xx items. Look for:
- Methods present in native but missing from the manifest (MG-xx items ŌĆö add them to the interface)
- Method parameter differences (e.g., native method takes additional filter params) ŌĆö match the native signature
- Stream/watch methods in native (e.g., Combine publishers, Flow collectors) ŌĆö add corresponding `Stream<T>` methods
- Batch operations in native (e.g., `updateMultiple`) ŌĆö add if present

### Use case decision logic

Generate a use case class only when one of the following is true:

1. The feature's manifest `feature_dependencies` is non-empty AND the manifest describes logic that combines data from both this feature's repository and a dep feature's repository
2. The manifest `migration_notes` or `docs/features/{feature}/NATIVE_PARITY.md` BL-xx items show logic that is duplicated across multiple screens/viewmodels

Otherwise: add a comment in the repository interface file:

```dart
// Use cases: none generated for this feature at this time.
// If cross-repository logic emerges during implementation,
// add use case classes to lib/use_cases/.
```

## Sub-agent A: Data layer detail

### Step 1: Discover pre-generated API client packages

Before writing any code, check whether a Swagger-generated client package already covers this feature's endpoints. The project generates full API client packages from Swagger specs (e.g. `shared/{project}_api_v2`, `shared/{project}_api_v3`, where `{project}` is the manifest's `project_name`). These packages already contain:
- All DTOs (response/request models as Dart classes)
- All remote method implementations (one method per endpoint)
- `json_serializable` annotations and generated `.g.dart` files

**Discovery procedure:**

1. Read the manifest's `api.swagger_source` field and `api_dependencies` endpoint list
2. Look at each endpoint's `path` and `method` ŌĆö note the API version in the path (e.g. `/v2/...`, `/v3/...`)
3. Scan `shared/` for packages with a name matching the API version pattern:
   ```
   shared/
   Ōö£ŌöĆŌöĆ {project}_api_v2/   ŌåÉ covers /v2/ endpoints
   Ōö£ŌöĆŌöĆ {project}_api_v3/   ŌåÉ covers /v3/ endpoints
   ŌööŌöĆŌöĆ ...
   ```
4. For each candidate package, read its barrel file (e.g. `shared/{project}_api_v2/lib/{project}_api_v2.dart` ŌåÆ `src/export.dart`) to build a capability map of available classes (client classes, DTO models)
5. For each endpoint in the manifest's `api_dependencies`, read the relevant client source file (e.g. `src/clients/users_remote_data_source.dart`) to check whether it contains a matching method with the correct HTTP verb and path

**Resolution outcome per endpoint:**

| Situation | Action |
|---|---|
| Method found in generated client | Use the generated DTO and client method directly ŌĆö do not regenerate |
| Method missing from generated client | Generate a stub, add a TODO (same stub convention as other missing symbols) |
| No generated client exists for this API version | Fall back to full manual generation (see fallback section below) |

**Mixed case (some endpoints covered, some not):** use the generated client for covered endpoints and generate stubs only for the gaps. Never regenerate a DTO or method that already exists in a generated package.

### Step 1b: Cross-reference with native data layer

After discovering generated client coverage, read `docs/features/{feature}/NATIVE_PARITY.md` and use the **DF-xx (Data Flow)** items to identify:

- **Caching patterns**: If the native repository caches API responses (in-memory or disk), note this for the Flutter repository. If caching is present in native, add a `// TODO: add caching (native uses {strategy})` comment in the repository implementation
- **Request construction**: If native code adds custom headers, query parameters, or request transformations beyond what the generated client provides, document these as TODOs or implement them in the repository
- **Pagination**: Match the native pagination pattern exactly (cursor vs offset, page size defaults, how "has more" is determined). NATIVE_PARITY.md DF-xx items should have the specifics ŌĆö use them for the repository method signatures
- **Error mapping**: If native code maps specific HTTP status codes to domain-specific errors (e.g., 409 ŌåÆ `ProfileAlreadyExistsError`), replicate this mapping in the Flutter repository's catch blocks

### Step 2: Read the domain scaffold

Read the committed domain scaffold before generating any data layer code:
- `features/{feature}/{feature}_domain/lib/{feature}_domain.dart` (barrel)
- All model files under `lib/models/`
- The repository interface file

### Step 3: Generate the mapper(s)

Regardless of whether endpoints are covered by a generated client or generated manually, the **mapper is always written by this sub-agent** ŌĆö it is the seam between the generated/external DTO world and the feature's own domain models.

Mappers are extension methods on the DTO class from the generated package:

```dart
// When using a pre-generated client (e.g. {project}_api_v2):
import 'package:{feature}_domain/{feature}_domain.dart';
import 'package:{project}_api_v2/{project}_api_v2.dart';  // generated client

extension {ModelName}ResponseMapper on {GeneratedDtoClass} {
  {ModelName} toDomain() {
    return {ModelName}(
      id: id,
      // map fields ŌĆö use the generated DTO's field names
      // check the generated class definition for exact names
    );
  }
}
```

```dart
// When using a manually generated DTO (fallback):
import 'package:{feature}_domain/{feature}_domain.dart';
import '../data_sources/{feature}_remote_data_source/dtos/{dto_name}.dart';

extension {ModelName}DtoMapper on {ModelName}Dto {
  {ModelName} toDomain() {
    return {ModelName}(
      {field}: {field},
    );
  }
}
```

Rules that apply in both cases:
- Never import from another feature's `_data` package ŌĆö duplicate minimal mapping logic if needed (see FFCA nested API objects pattern in FFCA_PATTERNS.md)
- If the generated DTO has fields not present in the domain model, silently drop them in the mapper
- If the domain model requires a field not present in the generated DTO, add a TODO on that field

### Step 4: Generate the repository implementation

The repository implementation changes shape depending on the discovery outcome:

**When using a pre-generated client:**

```dart
import 'package:{feature}_domain/{feature}_domain.dart';
import 'package:{project}_api_v{n}/{project}_api_v{n}.dart';
import '../mappers/{model}_mapper.dart';

class {Feature}sRepository implements I{Feature}sRepository {
  const {Feature}sRepository({
    required {Project}ApiV{n} apiClient,
  }) : _apiClient = apiClient;

  final {Project}ApiV{n} _apiClient;

  @override
  Future<{Model}> get{Model}ById(String id) async {
    final response = await _apiClient.{generatedMethodName}(id: id);
    return response.toDomain();
  }
}
```

**When using a manually generated remote data source (fallback):**

```dart
import 'package:{feature}_domain/{feature}_domain.dart';
import '../data_sources/{feature}_remote_data_source/{feature}_remote_data_source.dart';
import '../mappers/{model}_mapper.dart';

class {Feature}sRepository implements I{Feature}sRepository {
  const {Feature}sRepository({
    required {Feature}RemoteDataSource remoteDataSource,
  }) : _remoteDataSource = remoteDataSource;

  final {Feature}RemoteDataSource _remoteDataSource;

  @override
  Future<{Model}> get{Model}ById(String id) async {
    final dto = await _remoteDataSource.get{Model}ById(id);
    return dto.toDomain();
  }
}
```

In both cases: never return DTOs to callers. All public repository methods return domain models.

### Fallback: manual DTO and data source generation

Only used when **no generated client package covers the required endpoints**. Follow these rules:

**DTOs:**

```dart
import 'package:json_annotation/json_annotation.dart';

part '{dto_name}.g.dart';

@JsonSerializable()
class {ModelName}Dto {
  const {ModelName}Dto({
    required this.{field},
  });

  factory {ModelName}Dto.fromJson(Map<String, dynamic> json) =>
      _${ModelName}DtoFromJson(json);

  @JsonKey(name: '{json_key}')  // only if different from field name
  final {Type} {field};

  Map<String, dynamic> toJson() => _${ModelName}DtoToJson(this);
}
```

**Remote data source:**

```dart
class {Feature}RemoteDataSource {
  const {Feature}RemoteDataSource({required {ApiClient} apiClient})
      : _apiClient = apiClient;

  final {ApiClient} _apiClient;

  Future<{ModelName}Dto> get{ModelName}ById(String id) async {
    final response = await _apiClient.{endpointMethod}(id);
    return {ModelName}Dto.fromJson(response.data as Map<String, dynamic>);
  }
}
```

### Step 5: pubspec.yaml

Add the resolved generated client package(s) as dependencies (path or hosted, matching what the workspace already uses):

```yaml
dependencies:
  {feature}_domain:
    path: ../{feature}_domain
  {project}_api_v2:                # add only the versions this feature uses
    path: ../../../shared/{project}_api_v2
  # {project}_api_v3:             # add if feature uses V3 endpoints too
  #   path: ../../../shared/{project}_api_v3
  json_annotation: ^4.9.0        # only if manual DTOs were generated
```

Do not add `json_annotation` or `build_runner`/`json_serializable` to `dev_dependencies` if the feature data layer uses only pre-generated client DTOs ŌĆö those packages are already declared in the generated client package, not needed here.

### Step 6: Data layer test rules

Test scope adjusts based on what was generated:

- **Mapper tests**: always written ŌĆö pure unit tests, no mocks, test `dto.toDomain()` with known inputs including null/optional field cases; use the generated DTO class directly as input
- **Repository tests**: always written ŌĆö mock the API client (or data source if manual), assert domain model output and exception propagation
- **Remote data source tests**: only written if a manual data source was generated; if using a pre-generated client, the client's own tests cover transport behavior ŌĆö do not duplicate them
- Target 100%+ line coverage across all generated files (generated client files excluded from coverage calculation)

When mocking a pre-generated client, mock the generated class directly:

```dart
class Mock{Project}ApiV2 extends Mock implements {Project}ApiV2 {}
```



### Reading dependencies

Sub-agent B must read:
- The committed domain scaffold (models + repository interface + use cases if any)
- The ui_kit capability map (from barrel file exports)
- The manifest's screen/feature description and `parity_notes`
- **`docs/features/{feature}/NATIVE_PARITY.md`** ŌĆö this is the primary guide for generating correct cubits, states, and screen widgets
- **Figma design context** (if provided) ŌĆö per-screen `design_context` JSON containing layout, typography, colors, spacing, and `screenshot` for visual reference. Plus `file_key`/`node_id` for additional `get_metadata`/`get_design_context` MCP calls on child nodes

### Native parity requirements for presentation

NATIVE_PARITY.md contains numbered checklist items (BL-xx, ST-xx, VR-xx, UI-xx, NV-xx, DF-xx). Sub-agent B must address every item relevant to the presentation layer. For each item implemented, add a code comment: `// Ref: {ID}`

**State classes (ST-xx items):** For each screen, match the native ViewModel's observable states. Example ŌĆö if NATIVE_PARITY.md lists `ST-03: PersonalInfo has Saving state after updateProfile()`, generate a `PersonalInfoSaving` state subclass. Don't fall back to the generic three (loading/loaded/error) when native has more.

**Cubit methods (BL-xx items):** Map native ViewModel public methods 1:1 to cubit methods. If NATIVE_PARITY.md lists `BL-02: ViewModel has validateField() for per-field validation`, the cubit must have that method. Don't omit methods that seem minor ŌĆö they exist in native for a reason.

**Validation (VR-xx items):** Implement validation in the cubit (not the widget). Emit validation error states that the widget can display. Match the native behavior:
- Which fields are validated and when (on change, on submit, on blur)
- What error messages are shown for each validation failure
- Whether validation is field-level or form-level

**UI structure (UI-xx items):** Guide the widget tree:
- Match the component hierarchy (e.g., if native has a scrollable list with a header section and a menu section, build the same structure)
- Match conditional rendering (e.g., if native shows a banner after save, add that to the Flutter screen)
- Match user interactions (e.g., if native has pull-to-refresh on a list, add `RefreshIndicator`)

**Navigation (NV-xx items):** Determine:
- What parameters are passed when navigating to sub-screens
- Whether navigation is push (sub-screen) or replacement (tab switch)
- What happens when returning from a sub-screen (refresh parent data?)

### Figma design integration (FD-xx items)

Priority hierarchy for UI decisions:
1. FD-xx (Figma) ŌĆö visual appearance source of truth
2. UI-xx (native) ŌĆö behavioral source of truth
3. FFCA_PATTERNS.md ŌĆö code structure source of truth

When Figma data IS available:
- Layout: Figma auto-layout ŌåÆ Column/Row/Stack, fill ŌåÆ Expanded, hug ŌåÆ MainAxisSize.min
- Spacing: exact px values ŌåÆ AppSpacing tokens or `// Figma: {n}px` comments
- Colors: ŌåÆ Theme.of(context).colorScheme or ThemeExtension lookups
- Typography: ŌåÆ Theme.of(context).textTheme
- Cross-reference Figma component names with ui_kit exports
- Code comments: `// Ref: FD-xx`

When NOT available: fall back to UI-xx items. No warning needed.

Additional MCP calls: Sub-agent B may call `get_metadata` + `get_design_context` on child nodes if the screen-level response is truncated. Failures are non-blocking.

### BLoC vs Cubit decision

Use **Cubit** when:
- The screen has simple loading/loaded/error states with no discrete events
- There is only one primary action triggering state changes

Use **BLoC** (with explicit events) when:
- Multiple independent user actions trigger different state transitions
- The manifest describes complex interaction flows or the native analysis shows a ViewModel with multiple observed states

### State class rules

```dart
// For Cubit:
sealed class {Screen}State extends Equatable {
  const {Screen}State();
}

final class {Screen}Initial extends {Screen}State {
  const {Screen}Initial();
  @override
  List<Object?> get props => [];
}

final class {Screen}Loading extends {Screen}State {
  const {Screen}Loading();
  @override
  List<Object?> get props => [];
}

final class {Screen}Loaded extends {Screen}State {
  const {Screen}Loaded({required this.{model}});
  final {Model} {model};
  @override
  List<Object?> get props => [{model}];
}

final class {Screen}Error extends {Screen}State {
  const {Screen}Error({required this.message});
  final String message;
  @override
  List<Object?> get props => [message];
}
```

### Widget rules

- Screen widget: `StatelessWidget`, consumes BLoC via `BlocBuilder` or `BlocListener`
- Never instantiate BLoC/Cubit inside a widget ŌĆö BLoC is provided from the module level
- Use ui_kit components whenever a match is found in the capability map; read the component's source `.dart` file to get the exact constructor signature before using it
- When Figma `design_context` is available, cross-reference Figma component names with ui_kit exports. Use ui_kit match if found; otherwise build from primitives matching Figma layout data and add `// TODO: extract to ui_kit component matching Figma {component_name}`
- If a ui_kit component is needed but not found in the barrel exports: use a plain `Container` or `Text` placeholder, add a structured TODO immediately above

```dart
// TODO(vgv-migrate-implement): [{feature}_presentation] {ScreenName}
// Reason: ScCartItemRow not found in ui_kit barrel exports.
// Action: add ScCartItemRow to ui_kit, re-run with --serial to regenerate.
Container(color: Colors.red, child: const Text('ScCartItemRow placeholder')),
```

### Module file rules

Each screen folder exports a `{screen}_module.dart` that is the single public entry point:

```dart
export 'bloc/{screen}_cubit.dart';
export 'bloc/{screen}_state.dart';
export 'views/{screen}_screen.dart';
```

The feature barrel `{feature}_presentation.dart` exports each module file.

### Presentation test rules

- BLoC/Cubit tests: use `bloc_test` package, `blocTest<{Cubit}, {State}>(...)`
- Widget tests: pump the screen widget, mock the cubit, assert widget tree
- Provide a `helpers/pump_app.dart` with a `pumpApp` extension that wraps `WidgetTester` with necessary providers
- Target 100%+ line coverage

## Sub-agent C: Domain test detail

### What to test in domain

- Models: equality, hashCode, toString ŌĆö verify via `==` and `expect`
- Repository interface: write a `Fake{Feature}sRepository` that implements the interface with empty method bodies; verify it compiles and can be instantiated (this is a contract test)
- Use cases (if generated): mock both repositories, verify correct delegation and data assembly

### Domain test conventions

```dart
// Model test
void main() {
  group('{ModelName}', () {
    test('supports value equality', () {
      final a = {ModelName}(id: '1', ...);
      final b = {ModelName}(id: '1', ...);
      expect(a, equals(b));
    });

    test('props are correct', () {
      final model = {ModelName}(id: '1', ...);
      expect(model.hashCode, equals({ModelName}(id: '1', ...).hashCode));
    });
  });
}
```

## Phase 6: Merge worktrees detail

Each sub-agent runs in an isolated git worktree (via the Agent tool's `isolation: "worktree"` parameter). When a sub-agent completes with changes, it returns a worktree path and branch name.

Merge order and commands (adapt to project's git conventions from CLAUDE.md):

```bash
# Each sub-agent returns a branch name from its worktree
git merge --no-ff {data-worktree-branch} -m "feat: add {feature} data layer"
git merge --no-ff {presentation-worktree-branch} -m "feat: add {feature} presentation layer"
git merge --no-ff {domain-tests-worktree-branch} -m "test: add {feature} domain tests"
```

Worktrees are automatically cleaned up after merge if the sub-agent made changes, or immediately if no changes were made.

If any merge produces a conflict:
- Do not attempt to auto-resolve
- Print the conflicting files
- Halt with message: "Merge conflict detected. This indicates Phase 4 (workspace pubspec registration) may be incomplete or a sub-agent modified a shared file. Please resolve manually."

## Phase 9: Post-implementation review detail

### Detecting agent team availability

```bash

  echo "agent team mode"
else
  echo "sequential fallback mode"

```

Print the detected mode before starting any review work:

### Agent team mode: spawning the team

Spawn a team named `{feature}-review` with four reviewers. The team lead's role is synthesis only ŌĆö it does not perform its own review pass.

Spawn prompt template for the team lead:
```
Create a review team named {feature}-review for a newly implemented Flutter feature.

Spawn four reviewers with these exact roles:

Reviewer A (boundary): Review FFCA dependency boundaries across the three packages at
  features/{feature}/{feature}_domain
  features/{feature}/{feature}_data
  features/{feature}/{feature}_presentation
Check: presentation never imports from data, data never imports from presentation,
no cross-feature _data imports, stubs in lib/stubs/ not mixed with production code,
barrel files expose only intended public surface.
If you find a boundary violation that affects test imports, message Reviewer B with
the specific file and import path.

Reviewer B (coverage): Review test files across all three packages.
Check: every public class has a test file, mapper tests cover null/optional cases,
repository tests cover both success and exception paths, BLoC/Cubit tests cover all
state transitions, no domain or presentation tests import from {feature}_data,
no tests that only assert isA<SomeState>() without checking props.
If you find test mocks that reference an import flagged by Reviewer A, message
Reviewer A to correlate.

Reviewer C (stubs): Scan all TODO(vgv-migrate-implement) comments across all three packages.
Check: each stub has package/reason/action/tracking fields, identify stubs resolvable
with existing workspace packages, flag vague action fields, categorize by type.
If a stub is immediately resolvable and its current form may create a boundary issue,
message Reviewer A.

Reviewer D (native parity): Read docs/features/{feature}/NATIVE_PARITY.md and check each
numbered item against the generated code.
For BL-xx: verify business logic is implemented in cubit/repository.
For ST-xx: verify state transition exists in state class and cubit.
For VR-xx: verify validation rule is implemented with appropriate error states.
For UI-xx: verify UI pattern is present in screen widget.
For NV-xx: verify navigation pattern matches (push/replace, params, return behavior).
For DF-xx: verify data flow detail is in repository or cubit.
For MG-xx: verify manifest gap was addressed or has a TODO.
For FD-xx: verify Figma layout, spacing, and tokens are reflected in screen widget.
  Blocking: widget tree structure contradicts Figma layout (e.g., vertical vs horizontal).
  Action required: spacing/color significantly differs from Figma design.
  Informational: minor token rounding differences.
  If a Figma layout issue also affects test assertions, message Reviewer B.
If a missing state transition should be tested, message Reviewer B.

After all four complete, synthesize findings into docs/features/{feature}/REVIEW.md
using the format specified in the skill's SKILL.md Phase 9 section.
```

### Agent team mode: peer communication protocol

Reviewers communicate via the agent teams mailbox. Messages should be structured:

```
TO: Reviewer {A|B|C}
FROM: Reviewer {A|B|C}
FINDING: {one sentence description}
FILE: {path to relevant file}
IMPORT/SYMBOL: {specific import or symbol}
IMPLICATION FOR YOUR REVIEW: {one sentence}
```

The team lead does not relay these messages ŌĆö peer routing is direct.

### Agent team mode: synthesis

After all three reviewers report back, the team lead:
1. De-duplicates correlated findings (e.g., a boundary violation and its related test mock are one finding, not two)
2. Classifies each finding as blocking / action required / informational using the rules below
3. Writes `REVIEW.md`
4. Reports the summary to the main agent

### Sequential fallback mode

The main agent performs all four review passes in order: boundary ŌåÆ coverage ŌåÆ stubs ŌåÆ native parity. Each pass uses the same criteria as the corresponding reviewer above. After all four passes, write `REVIEW.md`.

The only difference from agent team mode: correlated findings (e.g., a boundary violation affecting a test) must be identified by the main agent during the synthesis step rather than via peer messages.

### Finding classification rules

| Classification | Criteria |
|---|---|
| **Blocking** | A direct FFCA violation (wrong import direction, cross-data import); test file missing for a public class that has no stub explanation |
| **Action required** | Shallow test assertions; vague stub action fields; boundary warning (not a hard violation but suspicious); stubs resolvable immediately |
| **Informational** | Barrel file exports internal class (worth noting but not a violation); test count below 100% estimate (not definitive without running coverage tool); manifest parity note not reflected in a code comment |

### Handling `--skip-review`

If `--skip-review` is set, skip Phase 9 entirely. Do not create `REVIEW.md`. Print:
```
[Phase 9] Skipped (--skip-review flag set).
Run /vgv-migrate-implement {feature-id} --skip-review=false to run review only.
```

Note: running the skill with `--skip-review=false` on an already-implemented feature (i.e., the packages already exist) should be treated as a review-only invocation ŌĆö skip Phases 2ŌĆō8, run only Phase 9 on the existing packages. This allows the review to be deferred and run separately.
